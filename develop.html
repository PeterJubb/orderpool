Development plan (overall site)

A pragmatic, buildable plan for shipping Order Pool end-to-end: public marketing, publisher submission, manufacturer acceptance, order management, and trust via public tracking.

v2 UI + flow-first MVP in phases Gated publishing + payments

North star outcome

What “working” means in one sentence.

Publishers

  • Create a brief in minutes (wizard-assisted)
  • Pay once, submit to pool, and get fast acceptance
  • Track progress with structured milestones
  • Share a status-only public link for trust

Manufacturers

  • Browse a clean quote pool (high-signal briefs)
  • Accept work without bidding chaos
  • Run orders with milestone updates + shipping
  • Reduce email overhead with structured updates

Platform pillars

The minimum capabilities the whole site depends on.

Core product

  • Auth + roles (publisher / manufacturer / admin)
  • Briefs (draft → wizard → submit)
  • Pool browsing + acceptance
  • Orders + milestones + updates
  • Files (private) + public tracking (status-only)

Trust & operations

  • Payments (pool posting fee)
  • Notifications (email + in-app)
  • Permissions + visibility rules
  • Audit history (who changed what)
  • Analytics (conversion + time-to-accept)

Phased roadmap

Build in the order that reduces risk and unlocks value fastest.

Phase Priority What you build Acceptance criteria
0. Prototype wiring Core v2 pages linked end-to-end + consistent header/footer + stable routes. (No backend required.) No broken links. No “publish” bypass: Draft → Wizard → Pay → Published only. Footer present on all pages (publisher/manufacturer).
1. Auth + roles Core Login/signup, role-based routing, session handling. Basic profile page. Users land in correct dashboard by role. Unauthorized routes redirect. Basic settings saved.
2. Briefs (draft) MVP Create/edit draft briefs, attachments metadata, draft list/detail. Draft can be created/edited. Attachments can be added/removed. Draft persists and renders correctly.
3. AI wizard MVP Wizard session, extraction, question flow, readiness checklist, structured output. Wizard asks only missing info, produces normalized JSON, readiness score gates payment. Session resumable.
4. Payments + submit MVP Stripe checkout/payment intent + server verification + submit to pool. Pool submission requires verified payment success. Webhook handling. Idempotent submit (safe retries).
5. Quote pool + acceptance MVP Manufacturer pool list + filters, quote detail, accept request → creates order. Manufacturers see matching briefs. Acceptance converts to order and removes/locks brief. Publisher notified.
6. Orders + milestones Core Order detail, milestone states, updates feed, shipping/tracking entry. Manufacturer can post updates + adjust milestone dates. Publisher sees timeline. Audit log recorded.
7. Public tracking Core Public order pages with status-only visibility rules. Public pages expose only safe fields (no pricing/files). Shareable URL works.
8. Notifications Scale Email + in-app notifications: acceptance, milestone changes, shipping updates, messages. Events trigger correct notifications with role-based recipients. User can configure notification preferences.
9. Admin + analytics Scale Admin moderation, category schemas, pricing config, reporting dashboards. Admin can tune required fields + fees without deploy. KPIs: completion, payment conversion, time-to-accept.
10. Nice-to-haves Later Messaging, vendor ratings, dispute flows, SLA templates, multi-currency, multi-warehouse shipping. Each feature has its own measurable KPI and doesn’t break MVP flows.

Data & permissions (must be crisp)

Avoid accidental leakage and keep “public tracking” trustworthy.

Primary objects

  • Users (role, org)
  • BriefDraft
  • WizardSession (questions/answers/extracted fields)
  • PoolBrief (published)
  • Order (after acceptance)
  • Milestones + Updates
  • Files (private)
  • PaymentIntent

Visibility rules

  • Public: order status + milestone labels only
  • Publisher: everything for their briefs/orders
  • Manufacturer: pool summary pre-accept; full details post-accept
  • Admin: full access + audit tooling

Definition of done (site-wide)

A checklist that keeps quality high as you scale pages.

Quality

  • No broken routes / missing footers
  • Consistent header by role
  • Responsive layout (mobile/tablet)
  • Accessible basics (labels, focus states)
  • Empty + full states for key lists

Security & ops

  • Role-based access enforced server-side
  • Payments verified server-side (webhook)
  • Audit logs for edits (milestones, shipping)
  • Rate limiting on wizard/chat endpoints
  • Error pages (404/500/maintenance)
